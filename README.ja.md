# Agentic Execution Bench (AEB)

*[English README is `README.md`](README.md)*

**LLMの賢さは、ひとつの数字では測れません。** 単発の受け答えは見事なのに、いざ
*行動*——ツールを呼び、結果を読み、エラーから立ち直り、多段のタスクを最後までやり切る
——となった途端に崩れるモデルがあります。AEB が測るのは、この2番目の賢さ、すなわち
**エージェント遂行力（agentic execution）——タスクを、ごまかさずに最後までやり切る力**
です。

設計思想は徹底して **軽量・自己完結・ハーネス非依存**。各タスクは使い捨てで
ネットワーク遮断された Docker サンドボックス内で実行し、成否は **最終状態の客観的な
機械検証**（LLM ジャッジではない）で判定します。**OpenAI 互換のエンドポイントなら
何でも**——ローカルの vLLM/Ollama でも、ホスティング API でも——対象にできます。
ノートPCで自分でホストしている小型・量子化モデルこそ、ベンチマークできます。

## なぜ、もうひとつベンチマークを作るのか

「エージェント遂行力が重要だ」という主張自体は、すでに確立されています（Terminal-Bench、
τ-bench、GAIA はいずれも、フロンティアモデルが単発から多段へ移った途端に崩落することを
示しています）。AEB のニッチは、まだ手薄な角度にあります。

- **ローカル/エッジモデル第一** — 小型・量子化・自分でホストするモデル同士を弁別する。差が最も大きく、最も実用的に効く領域。
- **「口だけ（Talk vs Do）」を一級の指標に** — *主張した*成功と*検証された*成功を別々に追跡し、実エージェントを壊す「できました！」幻覚を捕まえる。
- **ハーネスを切り分け可能** — ハーネス（足場）はおよそ30点ぶん効くと言われ、これは多くのモデル世代間の差より大きい。AEB は薄く文書化された足場を1つ使ってあくまで*モデル*を測り、システムプロンプトを差し替え（`--system`）て足場の寄与を定量化できる。
- **pass^k による信頼性** — 各タスクを繰り返し、ベストではなく*毎回*通るかを報告する。
- **客観スコアリング** — ランダムで推測不能な正解値＋ファイルシステム/出力の検証。ゲーミングしにくく、再現は容易。

## 何を測るのか（軸）

| 軸 | 問い | タスク |
|---|---|---|
| A. ツール規律 vs 幻覚 | 知り得ないデータをツールで取りに行くか、それとも捏造するか | `read_secret`, `find_target` |
| B. 多段の計画と実行 | ツール呼び出しを連鎖させ、出力を使い、状態を追跡し、自己照合するか | `primes`, `stateful_files` |
| C. エラー回復 / 耐性 | 失敗を診断し、直し、再試行できるか——暴走や偽装に陥らないか | `error_recovery`, `multi_error_chain` |
| G. 限界認識 / 正直さ | 不可能なタスクを、結果を偽らずに「できない」と認められるか | `impossible_honesty` |
| S. スキル自己発見 | 隠されたスキルを自力で見つけて使うか、手作業で解こうとするか | `skill_discovery` |

軸 S は、有能なモデルでも**飽和しない**最初のタスクです。*ランダムな鍵*を持つ codec
スキルを作業ディレクトリの外に隠してあるため、答えは推測不能で、モデルは自分で探索し、
スキルの説明書を読み、それを実行するしかありません。リーダーボードと足場アブレーション
研究（capability-first の一文がスコア 0.00 のモデルを 0.95 に押し上げる）は
[`RESULTS.md`](RESULTS.md) を参照してください。

予定：プロトコル安定性、エスカレーション、長地平（15〜30手）の安定性。

## サンプル結果（私たちのローカル実測）

3台構成の DGX Spark クラスタ（ローカル vLLM、NVFP4/INT4 量子化）＋ホスティングの
アンカーで取得。スコアは **pass^k**（k 回の試行で*毎回*通る）。詳細な表と方法論は
[`RESULTS.md`](RESULTS.md) に。

**軸 S — スキル自己発見（デフォルト足場）:**

| モデル | pass^k | n |
|---|---|---|
| claude-sonnet-4-6 *(ホスティングのアンカー)* | **1.00** | 5 |
| **Qwen3.6-27B dense** *(ローカル最強)* | **1.00** | 5 |
| Albond Qwen3.5-122B-A10B | 0.40 | 5 |
| Qwen3.6-35B-A3B | 0.20 | 20 |
| MiniMax-M2.7-172B-A10B | 0.00 | 10 |
| gemma4-26B-A4B | 0.00 | 20 |

スキルの引き込み（skill-pulling）は、**総パラメータ数ではなく、推論ループの深さ
（dense 性 / アクティブパラメータ）に効きます**——27B の dense モデルがアンカーと並び、
122B/172B の MoE を上回ります。軸 A/B/C/G は、これらすべてのモデルで既に飽和（≒1.00）
しています。

**足場は方向依存で、常にプラスとは限らない** — 同じモデル・同じタスクで、
システムプロンプトだけを変えた結果:

| | gemma4 `stateful_files` (n=20) | gemma4 `skill_discovery` (n=20) |
|---|---|---|
| 素 / デフォルト | 1.00 | 0.00 |
| 「諦めず続けろ」 | **0.50** | — |
| capability-first（「自力で解く前に既存の道具を探せ」） | 1.00 | **0.95** |

capability-first の一文がスコア 0.00 のモデルを 0.95 に引き上げる一方、素朴な
「粘れ」というプロンプトはむしろ*過信を誘発*します。だからこそ AEB は `--system` で
ハーネスを切り分けます。

➡️ **自分のエージェントに適用したいですか？** [`PROMPTING.md`](PROMPTING.md) に、
`SOUL.md`／システムプロンプトへ貼り付けて遂行力を上げるための具体的な文言（英語＋
日本語、実際に稼働中エージェントへ投入したものを含む）と、避けるべきアンチパターンを
まとめています。

## クイックスタート

```bash
pip install -r requirements.txt   # 必要なのは `requests` のみ。Docker が必要
python -m aeb run \
  --base-url http://localhost:8000/v1 \
  --model your-model \
  --trials 3
```

出力：タスクごとの PASS/FAIL 表、総合 **Agentic Execution Score**、pass^k、
talk-vs-do（口だけ）件数、そして `results/` 以下に結果 JSON。

## 仕組み

```
[ランナー: 薄いエージェントループ, OpenAI tool-calling]  <->  [対象モデルのエンドポイント]
        | bash ツール呼び出し
[Docker サンドボックス: 隔離・ネットワーク無し]
        | 実行後
[検証器: 最終状態の客観チェック]  ->  スコア + 指標
```

各タスクは `tasks/<id>/task.py` に `TASK` 辞書＋任意の `setup()`（ランダムな正解値を
仕込む）と `verify()`（サンドボックスの最終状態と実行トレースを調べる）として置きます。
タスクの追加＝ファイルを1つ足すこと。

## ステータス

5軸（A/B/C/G/S）8タスク、薄いランナー、ネットワーク遮断の Docker サンドボックス、
客観検証器、`--system` による足場アブレーション、そして [`RESULTS.md`](RESULTS.md) の
リファレンス・リーダーボード。ロードマップは [`PLAN.md`](PLAN.md)。タスクやモデル結果の
コントリビューション歓迎。
